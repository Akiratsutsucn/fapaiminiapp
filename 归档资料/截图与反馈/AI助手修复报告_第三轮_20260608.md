## 根本原因分析

### 为什么前两轮修复都失败了？

**第一轮修复尝试：** 添加 `@input="handleInput"` 事件处理器  
**结果：** ❌ 失败

**第二轮修复尝试：** 移除 `v-model`，改用 `:value` + `@change`  
**结果：** ❌ 失败

**问题根源：**
TDesign Vue Next 的 `t-textarea` 组件内部实现有问题。无论使用 `v-model`、`:value + @input` 还是 `:value + @change`，都会触发组件内部的类型检查代码：

```javascript
// TDesign 内部可能的实现（伪代码）
function handleInput(event) {
  if ('key' in event) {  // ❌ 当event是字符串时报错
    // 处理
  }
}
```

当输入框内容变化时，组件内部会检查事件对象，但传入的参数类型不符合预期，导致 `'key' in value` 报错。

**结论：** 只要使用 TDesign 的 `t-textarea` 组件，这个问题就无法避免。

---

## 第三轮修复方案

### 核心策略：完全绕过TDesign组件

**方案：** 使用原生HTML `<textarea>` 替换 `<t-textarea>`

### 修复1：替换为原生textarea

**文件：** `admin-web/src/views/ai/AiAssistantView.vue` (第79-87行)

**修改前（第二轮）：**
```vue
<t-textarea
  :value="inputMessage"
  placeholder="输入您的问题..."
  :autosize="{ minRows: 2, maxRows: 6 }"
  @keydown.enter.ctrl.prevent="onSendMessage"
  @change="handleTextareaChange"
  :disabled="streaming"
/>
```

**修改后（第三轮）：**
```vue
<textarea
  v-model="inputMessage"
  class="custom-textarea"
  placeholder="输入您的问题..."
  rows="3"
  @keydown.enter.ctrl.prevent="onSendMessage"
  :disabled="streaming"
></textarea>
```

**关键改动：**
- ✅ 使用原生 `<textarea>` 替代 `<t-textarea>`
- ✅ 恢复使用 `v-model`（原生textarea没有问题）
- ✅ 添加自定义样式类 `custom-textarea`
- ✅ 使用 `rows` 属性替代 `autosize`

### 修复4：增强会话创建验证

**文件：** `admin-web/src/views/ai/AiAssistantView.vue` (第145-164行)

**修改后：**
```javascript
async function onCreateSession() {
  try {
    const session = await createAiSession()
    console.log('创建会话成功:', session)

    // 验证session是否有效
    if (!session || !session.id) {
      console.error('创建的会话无效:', session)
      MessagePlugin.error('创建对话失败：返回数据无效')
      return
    }

    sessions.value.unshift(session)
    onSelectSession(session.id)
    MessagePlugin.success('已创建新对话')
  } catch (err) {
    console.error('创建对话失败:', err)
    MessagePlugin.error('创建对话失败')
  }
}
```

**改进点：**
- ✅ 添加创建成功的日志
- ✅ 验证返回的session对象和id字段
- ✅ 无效数据时提前返回，不加入列表
- ✅ 更详细的错误信息

## 技术对比

### 为什么原生textarea可以，TDesign不行？

| 特性 | 原生 `<textarea>` | TDesign `<t-textarea>` |
|------|------------------|----------------------|
| v-model支持 | ✅ 原生支持，无类型检查问题 | ❌ 内部实现有类型检查bug |
| 事件处理 | ✅ 标准DOM事件 | ❌ 封装的合成事件，有bug |
| 可控性 | ✅ 完全可控 | ❌ 受组件库实现限制 |
| 依赖 | ✅ 无外部依赖 | ❌ 依赖TDesign版本 |
| 调试难度 | ✅ 标准HTML，易调试 | ❌ 压缩代码，难定位 |

### 原生textarea的优势

1. **无类型检查bug** - 浏览器原生实现，经过充分测试
2. **更轻量** - 不需要额外的组件库代码
3. **更可控** - 完全掌控样式和行为
4. **兼容性好** - 所有现代浏览器都支持
5. **易维护** - 不受组件库版本更新影响

## 经验教训

### 1. 组件库的隐藏风险

使用第三方组件库时，要考虑：
- 组件实现是否稳定
- 遇到bug时能否快速修复
- 是否有替代方案

### 2. 调试压缩代码的困难

生产环境的错误信息：
```
TypeError: Cannot use 'in' operator to search for 'key' in 
    at r.<computed>.r.<computed> (index-D24QD1O2.js:18:11918)
```

压缩后的代码极难调试，定位问题花费大量时间。

### 3. 根因分析的重要性

前两轮修复都是"治标不治本"：
- 尝试拦截事件 → 但事件触发在更早阶段
- 更换事件类型 → 但组件内部实现仍有问题

第三轮直接替换组件 → 从根源消除问题

### 4. 渐进式降级策略

当高级组件有问题时：
1. 先尝试修复配置
2. 再尝试绕过问题点
3. 最后降级到原生实现

原生实现虽然功能简单，但稳定可靠。
